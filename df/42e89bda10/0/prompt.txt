## Type: `ValidationResult`

```typescript
type ValidationResult =
  | { valid: true }
  | { valid: false; reason: "invalid_format" | "disposable" | "undeliverable" };
```

### Invariants Identified

1. **Exclusive validity state**: A result is either valid or invalid -- never both, never neither.
2. **Reason presence tied to invalidity**: A `reason` field exists if and only if `valid` is `false`. Valid results carry no reason.
3. **Closed reason set**: The failure reason must be one of exactly three string literals, preventing arbitrary or misspelled error codes.
4. **Pipeline ordering convention** (implicit, not type-enforced): `validateEmail` applies checks in a specific order -- format, then disposable, then deliverability -- so the reason reflects the *first* failing layer.

### Ratings

- **Encapsulation**: 5/10

  The type itself is well-shaped, but the module's encapsulation has gaps. `isDisposableEmail` and `checkEmailDeliverability` are both exported, meaning consumers can call intermediate pipeline stages directly and bypass the ordering invariant. `isDisposableEmail` returns a raw `boolean` rather than a `ValidationResult`, creating two parallel "result vocabularies" for what is conceptually the same validation pipeline. A caller using `isDisposableEmail` directly gets no guidance that it should also check format first. The `TRUSTED_DOMAINS` set and `disposableSet` are module-private, which is good.

- **Invariant Expression**: 8/10

  This is the type's strongest quality. The discriminated union on `valid` is idiomatic TypeScript and works cleanly with narrowing. After checking `!validation.valid`, the compiler guarantees `reason` is accessible and is one of the three literals. The "reason only when invalid" constraint is structurally enforced -- you cannot construct `{ valid: true, reason: "disposable" }` without a type error. The literal string union for `reason` is self-documenting.

- **Invariant Usefulness**: 7/10

  The type prevents a real class of bugs: forgetting to handle failure reasons, accessing `reason` on a successful result, and passing arbitrary strings as reasons. However, the type does not capture *which email* was validated, so a `ValidationResult` floating in a variable has no connection back to its input. In a larger system this can lead to TOCTOU-style confusion where a result is applied to the wrong email. Additionally, the "fail open" behavior in `checkEmailDeliverability` (returning `{ valid: true }` on network errors) is invisible at the type level -- a caller cannot distinguish "genuinely validated" from "we could not check."

- **Invariant Enforcement**: 6/10

  Construction of `ValidationResult` values happens in three places (`validateEmail`, `checkEmailDeliverability`, and test code), and each constructs the objects inline. There is no single factory or constructor guarding creation, so nothing prevents a future contributor from returning `{ valid: false, reason: "typo" as any }`. The `validateEmail` function does validate its `email` input at runtime (null check, type check, `@` presence), but the format check (`email.includes("@")`) is extremely permissive -- strings like `"@@"` or `"@"` pass the disposable check path since `split("@")[1]` yields an empty string. The `isDisposableEmail` function silently returns `false` for malformed input rather than signaling an error, which is safe only because `validateEmail` checks format first -- but this ordering is not enforced if the function is called directly.

### Strengths

- **Textbook discriminated union**. The `valid` discriminant with conditional `reason` is exactly how TypeScript is meant to be used. Consumers get exhaustive narrowing for free.
- **Fail-open is a deliberate, documented choice**. The comments in `checkEmailDeliverability` make the policy explicit. This is a pragmatic decision for a signup flow.
- **The three-layer pipeline** (format, disposable, deliverability) is clean and easy to follow. Each check short-circuits, so expensive API calls only happen for emails that pass cheaper checks.
- **Literal string union for reasons** is far better than an open `string` or numeric error codes. It is self-documenting and exhaustively matchable.

### Concerns

1. **Leaky module surface**. Exporting `isDisposableEmail` and `checkEmailDeliverability` individually lets consumers bypass the pipeline ordering invariant. `isDisposableEmail` returns `boolean` instead of `ValidationResult`, creating an inconsistent contract. If these are exported for testing, a `@internal` convention or a dedicated test-only export would be more appropriate.

2. **Weak format validation**. The check `!email || typeof email !== "string" || !email.includes("@")` is the only format gate. It accepts clearly invalid strings like `"@"`, `"@@"`, `"foo@"`, and `"foo@bar@baz"`. The `split("@")[1]` extraction used downstream silently returns empty strings or wrong segments for these inputs. A basic regex (e.g., `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`) would close this gap without adding a heavy dependency.

3. **Untyped API response**. On line 58, `response.json()` returns `any`. The code trusts that `data.deliverability` is a string equaling `"UNDELIVERABLE"`. If the API changes its response shape, this silently fails open. A Zod schema or at least a type assertion would make the contract explicit.

4. **The `messages` record in `route.ts` duplicates the reason set without exhaustiveness checking**. It is typed as `Record<string, string>` rather than `Record<ValidationResult["reason"], string>` (extracting the reason from the invalid branch). This means if a fourth reason is added to `ValidationResult`, the compiler will not flag the missing message. The fallback `?? "Invalid email address"` masks this.

5. **No association between result and input**. The `ValidationResult` does not carry the email that was validated. In the current single-use confirm route this is fine, but if the type is reused for batch validation or passed across async boundaries, the detachment becomes a source of confusion.

6. **"Fail open" is invisible in the type**. A consumer cannot distinguish a `{ valid: true }` that means "we verified deliverability" from one that means "the API was down so we skipped the check." If this distinction ever matters (e.g., for audit logging or retry logic), the type will need to change.

### Recommended Improvements

**1. Tighten the module's public API.** Export only `validateEmail` and `ValidationResult`. If `isDisposableEmail` and `checkEmailDeliverability` need to be tested independently, export them from a separate `email-validation.internal.ts` or use the test file to import from the module internals directly (Bun supports this).

**2. Strengthen format validation.** Replace the `includes("@")` check with a minimal regex that ensures non-empty local part, non-empty domain, and at least one dot in the domain:

```typescript
const EMAIL_RE = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
```

This is not RFC 5322 complete, but it eliminates the most obvious garbage inputs without rejecting real addresses.

**3. Use the reason union to type the messages record in `route.ts`.** Extract the reason type and use it:

```typescript
type InvalidResult = Extract<ValidationResult, { valid: false }>;
const messages: Record<InvalidResult["reason"], string> = { ... };
```

This gives compile-time exhaustiveness. Remove the `?? "Invalid email address"` fallback so that a missing reason becomes a type error rather than a silent default.

**4. Add a lightweight schema for the Abstract API response.** Even a simple type guard suffices:

```typescript
function isUndeliverable(data: unknown): boolean {
  return typeof data === "object" && data !== null 
    && "deliverability" in data 
    && (data as { deliverability: unknown }).deliverability === "UNDELIVERABLE";
}
```

This makes the external contract explicit and prevents silent fail-open if the API response shape changes.

**5. Consider exporting `ValidationResult` as a named export.** Currently it is a module-scoped `type` that is not exported. Consumers in `route.ts` infer it from `validateEmail`'s return type, which works but means any code that wants to explicitly annotate a variable holding this result cannot do so. Adding `export type ValidationResult = ...` costs nothing and improves usability.
