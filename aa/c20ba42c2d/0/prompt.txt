about demo 5, the proof. how about this:
a validation rule enforced to coalesce null to a date (defaults to now). when the requirements changed, as features were built on top, the validation rule was not adjusted to reflect these new requirements. null values were no longer allowed to collapse down to a date. here's a written version with more details in German:

Der Vorfall
Im Januar 2026 wurden Kandidatenprofile unerwartet aus dem System gelöscht – obwohl die
betroffenen Mitarbeiter ihre Profile ordnungsgemäß verlängert hatten. Die Ursache war ein
technischer Fehler in der Datenverarbeitung.
Was technisch passierte: Bei bestimmten Aktualisierungen wurde das Erstellungsdatum nicht
mitgesendet. Das System hätte das bestehende Datum beibehalten sollen. Stattdessen
wandelte eine Validierungsregel das fehlende Datum in den 1. Januar 1970 um – ein Datum, das
mehr als 50 Jahre in der Vergangenheit liegt.
Das automatische Löschsystem erkannte diese Profile als „älter als 12 Monate" und markierte
sie zur Löschung. Die betroffenen Profile wurden physisch aus der Datenbank entfernt.
Der Fehler: Leeres Datum wurde zu 1970
Eingabe:
(leer)
Fehlerhafte
Umwandlung
z.coerce.date()
Ergebnis:
01.01.1970
Das System interpretierte Profile als 50+ Jahre alt - Automatische Löschung
Auswirkung: Kandidatenprofile wurden physisch aus der Datenbank gelöscht.
Diese Datewiederhergestellt werden.
ProfileHub - Technischer Vorfallbericht | Seite 3
Ursachenanalyse
Wie bereits erwähnt, wurde die Profilverlängerung nachträglich zum System hinzugefügt. Um zu
verstehen, warum der Fehler entstehen konnte, muss man die ursprüngliche Systemarchitektur
betrachten.
Die ursprüngliche Schutzregel:
In der ersten Version des ProfileHubs wurde jedes Profil genau einmal erstellt und erhielt dabei
automatisch ein Erstellungsdatum. Dieses Datum durfte niemals leer sein – eine
Validierungsregel stellte dies sicher. Fehlte das Datum aus irgendeinem Grund, wurde es
automatisch auf einen Standardwert gesetzt. Diese Regel war sinnvoll, denn ohne
Erstellungsdatum hätte das automatische Löschsystem nicht funktioniert.
Warum die Erweiterung das Problem verursachte:
Mit der Profilverlängerung änderten sich die Anforderungen grundlegend. Bei Aktualisierungen
wird das Erstellungsdatum bewusst nicht mitgesendet – das System soll das bestehende Datum
beibehalten. Die alte Validierungsregel inieses „nicht gesendet" jedoch als „leer"
und setzte den Standardwert: 01.01.1970.
Warum der Fehler nicht früher erkannt wurde:
• Es gab keine automatisierten Tests, die das Verhalten der Validierungsregel
dokumentierten
• Ohne Tests war nicht erkennbar, dass die Erweiterung bestehende Regeln beeinflusste
• Die Löschung erfolgte zeitverzögert durch einen Hintergrundprozess – nicht sofort sichtbar
• Der Zusammenhang zwischen Validierung und Löschung war nicht offensichtlich
Kernerkenntnis: Automatisierte Tests machen Verhaltensänderungen sichtbar,
die bei Systemerweiterungen sonst unbemerkt bleiben können.
Ergriffene Maßnahmen
1. Sofortmaßnahme: Korrektur der Validierungsregel
Die problematische Validierungsregel wurde durch eine angepasste Version ersetzt. Das System
unterscheidet nun zwischen „Datum nicht gesendet" (beibehalten) und „Datum explizit leer"
(Fehler). Fehlende Werte werden nicht mehr automatisch umgewandelt, sondern das
bestehende Datum bleibt erhalten.
ProfileHub - Te Seite 4
Die Lösung: Korrekte Behandlung leerer Daten
Eingabe:
(leer)
Prüfung:
Ist Wert leer?
Ja = Behalte leer
OK
2. Langfristige Maßnahme: Test-Infrastruktur
Um künftige Fehler dieser Art zu verhindern, wurde eine umfassende Test-Infrastruktur mit 263
automatisierten Tests eingeführt. Diese Tests dokumentieren das erwartete Verhalten aller
kritischen Funktionen und werden automatisch bei jeder Codeänderung ausgeführt.
