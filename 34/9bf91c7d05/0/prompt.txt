## Type: `ValidationResult`

```typescript
export type ValidationResult =
  | { valid: true }
  | { valid: false; reason: "invalid_format" | "disposable" | "undeliverable" };
```

### Invariants Identified

1. **Discriminant completeness**: When `valid` is `true`, no `reason` field exists. When `valid` is `false`, a `reason` must be present and must be one of exactly three string literals.
2. **Reason exclusivity**: A failure can only be attributed to one of three causes -- format, disposability, or deliverability -- matching the three validation layers in the pipeline.
3. **No partial/ambiguous state**: There is no "maybe valid" or "pending" state. The result is terminal.

### Ratings

- **Encapsulation**: 7/10
  This is a plain discriminated union -- a data-only type with no behavior attached. That is appropriate here because it is a return type from a pure validation pipeline, not a domain entity with mutation. The type itself cannot be "violated" since TypeScript enforces the union at every use site. It loses points because the `reason` string literal union is defined inline rather than extracted, which means any consumer that wants to enumerate reasons (as `route.ts` does with `VALIDATION_MESSAGES`) must use the somewhat awkward `Extract<ValidationResult, { valid: false }>["reason"]` utility type. A named export for the reason union would improve ergonomics.

- **Invariant Expression**: 9/10
  The discriminated union pattern is the gold standard for expressing "success or failure with context" in TypeScript. The `valid` discriminant makes narrowing automatic: after checking `!validation.valid`, TypeScript knows `reason` is available. The literal string union for `reason` is self-documenting -- you can read the type and immediately understand the three failure modes. The only gap is that the relationship between each `reason` and its corresponding pipeline layer is implicit (you need to read the function body to know which layer produces which reason).

- **Invariant Usefulness**: 8/10
  The type prevents several real bugs. The exhaustive `reason` union means the `VALIDATION_MESSAGES` record in `route.ts` will fail to compile if a new reason is added without a corresponding message -- that is excellent. The absence of `reason` on the success branch prevents accidentally reading a stale or meaningless reason from a passing result. However, the type does not carry any additional context that might be useful downstream (e.g., the normalized email, or which API call was skipped). Whether that is a concern depends on future needs; for now, the type is well-scoped.

- **Invariant Enforcement**: 7/10
  Since this is a return type (not a constructable class), enforcement depends on the functions that produce it. All three production paths in `validateEmail` and `checkEmailDeliverability` return correctly shaped objects, and TypeScript's return-type annotation ensures this at compile time. The weakness is on the input side: `validateEmail` accepts a raw `string`, performs runtime validation, and then returns a `ValidationResult`. There is no branded/opaque type flowing out of the success case, so downstream code (the `POST` handler) proceeds to use the original raw `email` string for database writes and email sending. A validated email branded type would close this gap, though it would add complexity.

### Strengths

- **Textbook discriminated union.** The `valid` boolean discriminant gives free narrowing in every consumer. The `route.ts` handler demonstrates this perfectly on line 32: after `!validation.valid`, it accesses `validation.reason` without any cast.
- **Exhaustive reason mapping.** The `VALIDATION_MESSAGES` record type `Record<Extract<ValidationResult, { valid: false }>["reason"], string>` is a compile-time guarantee that every failure reason has a user-facing message. Adding a fourth reason to the union will immediately surface a type error.
- **Fail-open semantics are consistently typed.** Every error/fallback path in `checkEmailDeliverability` returns `{ valid: true }`, which is a deliberate product decision (don't block signups on API failures). The type makes this visible and auditable.

### Concerns

1. **Inline reason union.** The reason literals are embedded inside `ValidationResult` rather than extracted as a named type. This forces consumers to use `Extract` gymnastics (as seen on line 14 of `route.ts`) to reference the reason set independently. If a second consumer needs the reason type, this pattern will be duplicated.

2. **No validated email flowing out.** After `validateEmail` returns `{ valid: true }`, the caller still works with the original unvalidated `string`. There is nothing in the type system preventing the caller from accidentally using a different variable, or skipping validation entirely and passing any string to the database. A branded type like `type ValidatedEmail = string & { __brand: "ValidatedEmail" }` on the success branch would make "must validate before use" enforceable at compile time.

3. **`isDisposableEmail` returns `boolean`, not `ValidationResult`.** This function is exported and used independently, but it returns a plain boolean rather than participating in the `ValidationResult` type. This creates two parallel "validation result" representations in the public API. If a consumer calls `isDisposableEmail` directly, they get no structured reason -- they must construct their own result.

4. **Test accesses `reason` without narrowing.** On lines 33, 45, and 51-52 of the test file, `result.reason` is accessed directly after asserting `result.valid` is `false` via `expect`. TypeScript does not narrow based on test assertions, so these lines rely on `@ts-expect-error`-style looseness or Bun's lack of strict type checking in test files. This is minor but worth noting.

### Recommended Improvements

1. **Extract the reason union as a named type.**
   ```typescript
   export type ValidationFailureReason = "invalid_format" | "disposable" | "undeliverable";
   
   export type ValidationResult =
     | { valid: true }
     | { valid: false; reason: ValidationFailureReason };
   ```
   This eliminates the `Extract` utility type usage in `route.ts` and gives consumers a clean import.

2. **Consider a branded email type on the success path.** If this codebase grows to have more endpoints that require validated emails, returning `{ valid: true; email: ValidatedEmail }` from `validateEmail` would let you encode "this email has passed validation" in the type system. The success branch would carry the normalized email, and downstream functions could require `ValidatedEmail` as input. This is a moderate complexity addition -- worth it if validation becomes a cross-cutting concern, overkill if this is the only call site.

3. **Narrow `isDisposableEmail` to an internal helper or align its return type.** Since it is exported, consider whether external consumers actually need it. If they do, consider having it return a `ValidationResult` for consistency. If they do not, un-export it to reduce the public API surface and eliminate the parallel boolean-vs-union result representation.
